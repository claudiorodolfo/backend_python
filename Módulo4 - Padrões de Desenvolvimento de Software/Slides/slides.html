Recapitulação - Padrões de Projeto

Soluções comprovadas para problemas recorrentes em desenvolvimento de software.

  • Soluções reutilizáveis e testadas
  • Linguagem comum entre desenvolvedores
  • Facilitam comunicação e documentação
  • Não são fórmulas, são guias adaptáveis
  • Usar quando fizer sentido, não forçar uso
Slide 36 de 54

Recapitulação - Padrões de Criação

Padrões que tratam da criação de objetos.

  • Singleton: instância única
  • Factory: criar objetos sem especificar classe exata
  • Builder: construção passo a passo
  • Prototype: clonar objetos existentes
  • Object Pool: reutilizar objetos caros
Slide 37 de 54

Recapitulação - Padrões Estruturais

Padrões que facilitam o design mostrando como compor objetos.

  • Adapter: adaptar interface incompatível
  • Decorator: adicionar comportamentos dinamicamente
  • Facade: interface simplificada para subsistema
  • Proxy: controlar acesso a objeto
  • Bridge: separar abstração de implementação
  • Composite: tratar objetos individuais e grupos uniformemente
Slide 38 de 54

Recapitulação - Padrões Comportamentais

Padrões que tratam de comunicação entre objetos.

  • Observer: notificar mudanças
  • Strategy: algoritmos intercambiáveis
  • Command: encapsular requisições
  • Template Method: definir esqueleto de algoritmo
  • Chain of Responsibility: passar requisição por cadeia
Slide 39 de 54

Recapitulação - Padrões Arquiteturais

Padrões de alto nível para estruturação de aplicações.

  • MVC: Model-View-Controller separa responsabilidades
  • Repository: abstração de acesso a dados
  • Service Layer: lógica de negócio centralizada
  • Dependency Injection: injeção de dependências
  • Facilitam manutenção e testes
Slide 40 de 54

Recapitulação - SOLID: Single Responsibility

Uma classe deve ter apenas uma razão para mudar.

  • Responsabilidade única e bem definida
  • Facilita manutenção e testes
  • Reduz acoplamento
  • Classes pequenas e focadas
  • Evitar "god classes"
Slide 41 de 54

Recapitulação - SOLID: Open/Closed

Aberto para extensão, fechado para modificação.

  • Estender comportamento sem modificar código existente
  • Usar herança e composição
  • Polimorfismo e abstrações
  • Mantém código estável
  • Facilita adição de features
Slide 42 de 54

Recapitulação - SOLID: Liskov Substitution

Objetos de superclasse devem ser substituíveis por objetos de subclasse.

  • Subclasses não devem quebrar comportamento esperado
  • Contratos e pré/pós condições
  • Herança deve fazer sentido semântico
  • Evitar herança por conveniência
  • Garantir substitutibilidade
Slide 43 de 54

Recapitulação - SOLID: Interface Segregation

Muitas interfaces específicas são melhores que uma interface geral.

  • Interfaces pequenas e focadas
  • Clientes não devem depender do que não usam
  • Evitar interfaces "gordurosas"
  • Separar por responsabilidade
  • Facilita implementação
Slide 44 de 54

Recapitulação - SOLID: Dependency Inversion

Depender de abstrações, não de implementações concretas.

  • Módulos de alto nível não devem depender de baixo nível
  • Ambos devem depender de abstrações
  • Inversão de controle (IoC)
  • Dependency Injection
  • Facilita testes e manutenção
Slide 45 de 54

Recapitulação - Quando Usar Padrões

Padrões devem ser usados com sabedoria.

  • Entender o problema primeiro
  • Padrão deve simplificar, não complicar
  • Evitar over-engineering
  • Começar simples, refatorar quando necessário
  • Comunicar intenção claramente
Slide 46 de 54

Recapitulação - Anti-padrões

Padrões a evitar em desenvolvimento.

  • God Object: classe que faz tudo
  • Spaghetti Code: código desorganizado
  • Copy-Paste Programming: duplicação
  • Golden Hammer: usar mesmo padrão sempre
  • Premature Optimization: otimizar antes de medir
Slide 47 de 54

Recapitulação - Refatoração para Padrões

Como aplicar padrões em código existente.

  • Identificar problemas recorrentes
  • Refatorar gradualmente
  • Manter testes passando
  • Documentar decisões
  • Medir impacto antes e depois

Recapitulação - Padrões em Python

Características específicas ao aplicar padrões em Python.

  • Duck typing reduz necessidade de interfaces explícitas
  • Decorators facilitam implementação
  • Context managers para gerenciamento de recursos
  • Generators para padrões como Iterator
  • Aproveitar características da linguagem

Recapitulação - Combinando Padrões

Padrões frequentemente trabalham juntos.

  • Repository + Dependency Injection
  • Factory + Strategy
  • Observer + Command
  • Facade + Adapter
  • Entender relações entre padrões

Como padrões facilitam testes.

  • Interfaces facilitam test doubles
  • Separação de responsabilidades
  • Testes isolados e rápidos
  • Cobertura de código facilitada
